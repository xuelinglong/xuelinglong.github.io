深拷贝与浅拷贝

指的是对于Object、Array这种引用类型，当从一个变量复制引用类型的值时，复制的其实是一个指针，两个变量指向同一个堆对象，改变其中一个变量，另一个也会受到影响。

浅拷贝：拷贝引用。

深拷贝：拷贝实例(新的实例与之前的实例互不影响)。

浅拷贝包括：Array.prototype.slice()，Array.prototype.concat()，

深拷贝常用方法：JSON.parse(JSON.stringify(obj))、fon…in加递归

JOSN对象中的stringify可以把一个js对象序列化为一个JSON字符串，parse可以把JSON字符串反序列化为一个js对象，通过这两个方法，也可以实现对象的深复制。

Promise的finally、all、race等方法

finally是不管promise对象最后状态如何，都会执行的操作;finally方法的回调函数不接收任何参数，所以finally里面的操作是与promise状态无关的。

all用于将多个promise实例包装成一个新的promise实例；接受一个数组（内部是一个个的promise实例，若不是则会调用resolve方法转换为promise实例再处理）为参数；只要有一个promise被rejected那么p就会变成rejected状态（第一个被rejected的promise实例的返回值会传递给p的回调函数），必须所有的promise都fulfilled那么p才会fulfilled（所有promise实例的返回值组成一个数组传递给p的回调函数）。

const p = Promise.all([p1, p2, p3])

race也用于将多个promise实例包装成一个新的promise实例；接受一个数组（内部是一个个的promise实例，若不是则会调用resolve方法转换为promise实例再处理）为参数；只要有一个promise改变状态那么p就会改变状态，且率先改变状态的promise实例的返回值会传递给p；

const p = Promise.race([p1, p2, p3]);

all与race的相同点：

  1. 都用于将多个promise实例包装成一个新的promise实例；
  2. 如果参数数组中有非promise实例的存在，都会先调用Promise.resolve方法转换为promise；

all与race的区别：

  1. 状态的改变方式不同
  2. 传递给回调函数的值不同

跨域的方法

同源需满足：域名相同 + 协议相同 + 端口相同。否则跨域请求的返回结果会被浏览器拦截掉。

跨域限制以下几种行为：cookie、localStorage无法读取；DOM 和 Js对象无法获得；AJAX 请求不能发送。

  • jsonp跨域（只支持get类型请求可用）：利用script标签的src属性实现跨域，服务器解析请求的URL(拼接的有约定callback)，然后把约定的回调函数的参数返回给客户端，客户端执行约定好的回调函数。

jsonp实现过程解析：客户端注册一个callback如fn，通过script加载静态资源，拼接上callback=fn，服务器得到callback的名字是fn后，使用fn(params)的方式把要输出的内容返回给客户端，然后客户端执行预先定义好的fn函数。

  • postMessage跨域：otherwindow.postMessage(data信息对象，如JSON.stringify(data),origin目标域名，如'http://www.domain2.com')向其他窗口发送信息；监听窗口对象的message事件接收从其他窗口发送过来的信息window.addEventListener(“message”, function(e){console.log(e);},false)。完整流程是a通过postMessage向b发送信息，b监听message事件接收信息然后处理之后通过postMessage给a返回信息，然后a监听message事件接收b回传的数据。

postMessage使用场景：页面和其打开的新窗口的数据传递；多窗口之间消息传递；页面与嵌套的iframe消息传递。

  • window.name跨域：a页面引入一个iframe，把b设置为与a在同一域名下，然后通过iframe.contentWindow.name获取数据，b页面中把数据放在window.name=”传递的信息”。
  • vue等可以配置proxy代理跨域。

缓存方案

强缓存：Cache-Control：max-age定义了一个最大使用期。

协商缓存：需要和服务器进行协商是否使用本地缓存。

浏览器缓存

1.localstorage：可无视用户主动刷新页面，并且可储存较大体积的资源(2M以上)，setItem存getItem取。

2.sessionStorage：关闭窗口后失效，setItem存getItem取。

场景：页面滚动，将滚动位置存到session中 -> 再次进到页面中，到session中取出上次保存的浏览位置 -> 滚动到对应位置。

http协议、contentType

TCP/IP可分为四层：

  • 应用层：协议位于这一层，用于向用户提供应用服务。
  • 传输层：为应用层实体提供端到端的通信功能。
  • 网络层：规定了通过什么样子的路径到达对方计算机，并把数据包传送给对方。
  • 网络接口层：负责接收IP数据包并通过网络发送，或者从网络上接收物理帧，抽出IP数据包，交给网络层。

建立连接TCP协议三次握手：发送端发送一个带SYN标志的数据包给接收端，接收端收到后回传一个带有SYN/ACK标志的数据包表示传达确认信息，发送端再回传一个带ACK标志的数据包，代表“握手”结束。

断开连接四次分手：TCP协议是全双工模式（同时进行双向的数据传输），即发送端也可以是接收端，发送端发送断开请求，接收端确认断开请求，两次分手，两个连接总共是四次分手。

URI统一资源标识符，URL统一资源定位符，URL的范围大于URI，虽然很多AJAX工具的地址参数是URL，但HTTP请求的地址其实是URI。

首部压缩：每个http请求都有一个请求首部存放如cookie、useragent等重要信息字段，每次请求这些字段都是一样的却还必须要带上就很浪费，为了优化引入了首部压缩机制，客户端和服务端共同维护同一张首部信息表，每次请求只发送索引号就可以了。

cookie管理状态：没有cookie信息状态下，客户端发送一个验证给服务端，服务端验证通过后将验证信息添加到cookie中，将其返回给客户端，客户端拿到cookie之后储存在本地，之后客户端再次请求时带上本地的cookie，服务器根据带过来的cookie验证信息进行验证，验证通过直接返回数据，未通过可跳转到登录页面。

HTTP报文大致可分为请求行/响应行、报文首部和报文主体两块。

  • 请求行主要包括：请求方法、URI地址和HTTP协议版本号。
  • 响应行主要包括：HTTP协议版本号、HTTP状态码和状态描述。

请求报文首部字段：

  • Accept：可处理的媒体类型
  • Accept-Charset：可接收的字符集
  • Accept-Encoding：可接受的内容编码
  • Accept-Language：可接受的自然语言

相应报文首部：

  • Accept-Ranges：可接受的字节范围
  • Location：令客户端重新定向到的URI
  • Server：HTTP服务器的安装信息

实体首部字段：

  • Content-Type：实体主类的类型
  • Content-Encoding：实体主体适用的编码方式
  • Content-Language：实体主体的自然语言
  • Content-Length：实体主体的的字节数
  • Content-Range：实体主体的位置范围，一般用于发出部分请求时使用。

请求方法

  • get：请求服务器上的某一资源。
  • post：向指定资源提交数据进行处理请求，数据包含在请求体中。
  • put：用来传输文件，将文件内容放进报文主体中，保存到URI指定位置上。
  • delete：请求URI删除指定资源。
  • options：查询相应URI支持的HTTP方法。
  • head：获得报文首部，一般用于验证URI是否有效。与GET方法类似，只是不返回报文主体。

get与post请求区别

  • get重点用于从服务器获取数据，post重点用于向服务器发送数据；
  • get参数拼接在URL上面，用户可见安全性低，post参数包含在请求头里面，用户不可见安全性高。
  • get传输数据量小受URL长度限制但是效率高，post可传输大量数据。
  • get只能支持ASCII字符，post支持标准字符集，可正确传输中文字符。

常见的状态码

  • 100(继续)：临时响应通知客户端它的部分请求已经被服务器接收，且仍未被拒绝，客户端应当继续发送请求。
  • 200(成功)：正常处理。
  • 204(无内容)：请求被受理但没有资源可以返回。
  • 206(部分内容)：服务器成功执行了客户端的范围请求，响应报文中包含由Content-Range指定范围的实体内容。
  • 301(永久重定向)：请求的资源已被分配了新的URI，以后应使用资源现在所指的URI。响应报文的首部字段会用location字段来标记新的URI。
  • 302(临时重定向)：本次请求的资源被临时分配了新的URI，客户端应当继续向原有地址发送以后的请求。响应报文的首部字段会用location字段来标记新的URI。
  • 303(查看其他位置)：当前请求的响应可以在另一个URI上被找到，客户端应当采用GET的方式访问那个资源，响应报文的首部字段location中也会标记本次被分配到的新的URI。
  • 304(未修改)：不会返回任何资源，客户端通常会缓存访问过的资源，请求报文需要提供一个头信息指出客户端希望只返回在指定日期之后修改的资源。
  • 307(临时重定向)：只能使用post方法。
  • 400(请求错误)：前后端参数没有协同好。
  • 401(需要认证)：请求需要认证。
  • 403(被禁止)：请求被拒绝。
  • 404(未找到)：没有找到这个资源。
  • 405(方法错误)：请求方法错误，响应报文中返回一个Allow头信息用以表示出当前资源能够接受的请求方法的列表。
  • 500(服务器内部错误)：服务器在执行请求时发生了错误。
  • 503(服务不可用)：服务器现在无法处理请求。

防抖是什么

场景描述：用户拖拽改变窗口大小过程中窗口的大小一直在改变，如果在resize事件中绑定函数，这个函数将会一直触发，无意义且会浪费资源，此时可用函数防抖来优化。

防抖就是在短时间内多次触发同一个函数，只执行最后一次，或者只在开始时执行。

特性：触发了一个事件后：如果有一个定时任务待执行，就清除定时器，重新计时；如果没有任务待执行，就定时执行这个事件。

正则表达式

==与===的区别

==抽象相等，比较时先进行类型转换然后再比较值。

===严格相等，会比较两个值的类型和值。

===对比模式：

  • 对比Type(a)和Type(b)，不相等返回false，
  • 相等判断Type(a)若是’undefined、Null’直接返回true，
  • 若是String则对比长度字符序列完全相同才返回true，
  • Type(a)若是Boolean则a,b都是true或者都是false才会返回true，
  • 若Type(a)是Number则a,b任意一个是NaN就返回false，数字的话数字值相等返回true，+0与-0的话返回true，其他返回false。

==对比模式：

  • 对比Type(a)和Type(b)，相等返回a===b的结果，不同进入下面判断
  • 若是一个undefined一个null则返回true，
  • 若一个是Number一个是String，则返回a==ToNumber(b);
  • 若Type(a)或者Type(b)是Boolean则返回ToNumber(a) == b / a == ToNumber(b)
  • 如果Type(a)是String或Number或Symbol中的一种并且Type(b)是Object，返回 a==b的原始值 的结果,
  • 如果Type(a)是Object并且Type(b)是String或Number或Symbol中的一种，返回 a的原始值==y 的结果
  • 其它返回false。

输入一个URL到页面渲染的过程

  1. DNS查询：浏览器根据DNS服务器得到域名的IP地址；
  2. TCP连接：
  3. HTTP请求即响应：向这个IP的机器发送HTTP请求；
  4. 服务器响应：服务器收到、处理并返回HTTP请求，
  5. 客户端渲染：浏览器得到返回内容，然后处理HTML标记构建DOM树，处理CSS标记构建CSSOM树，将DOM与CSSOM树合并成一个渲染树(Rendering Tree)，根据渲染树来布局、计算每个节点的几何信息，最后将各个节点绘制到屏幕上。

性能优化的方案

网络方面：减少http的请求次数，比如将多个文件打包成一个；减少资源文件的体积；缓存

渲染和操作DOM方面：将css文件放在头部，js文件放在底部；引入顺序上，CSS资源先于JavaScript资源；减少对DOM的直接操作；减少定时器的使用；

数据方面：图片的预加载；首屏加载进度条的显示；使用JSON进行数据交互，对部分常用数据进行缓存，数据埋点和统计。

渲染过程中构建DOM时，遇到script脚本会停止构建DOM树而执行JS脚本，然后等JavaScript引擎运行完毕，浏览器再从中断的地方恢复DOM构建，因此想要首屏渲染快速的话建议把script标签放在body底部，或者添加defer或者async属性（async与defer属性对于 inline-script都是无效的，只针对设置了src属性的script标签）。

因为js文件的影响，CSSOM也会阻塞DOM的构建过程，原本CSSOM与DOM互不影响，但是JS不仅可以更改DOM，也可以更改CSS，且JS更改CSS时必须要拿到完整的CSSOM(CSSOM不完整就不能使用)，所以就会导致若浏览器还没有完成CSSOM的下载和构建的话，JS脚本的执行和DOM构建就会被延迟，顺序是：先下载和构建CSSOM -> 执行js脚本 -> 构建DOM。

无属性script与defer与async的区别：

无属性的script遇到会直接执行；async表示异步执行引入的JavaScript，加载好以后会立即执行，依旧会阻塞load事件；defer表示延迟执行引入的JavaScript，即加载时不中断HTML的解析，当全部document解析完毕且加载完所有defer-script之后触发DOMContentLoaded事件，在加载多个js脚本的时候，defer是有顺序的。
